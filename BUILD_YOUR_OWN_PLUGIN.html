Building a Claude Code Plugin BEEF CO · PLATFORM ENGINEERING
A plugin is a folder of files that teaches Claude Code a workflow your team repeats. A great plugin isn't just a pile of skills. The useful ones turn skills, agents, and hooks into a repeatable workflow. You've seen adoption-os find the patterns in how your team uses Claude Code. Here's how to build one for a workflow of your own.
adoption-os pattern Hook observes → Agent analyzes → Skill produces the file/output → README proves it works

1Start with one repeated task

Do not build a general productivity plugin. Build one plugin for one painful, repeated workflow.

2Know the four building blocks

A plugin is assembled from four kinds of part:

  • SKILLThe command and output template.
  • AGENTLooks through messy inputs to do work a person now does by hand.
  • HOOKSomething that runs automatically when Claude does something.
  • MCPA connection to an external system like GitHub, Jira, Slack, or Datadog.

Your first plugin needs just one, usually a Skill. The loop at the top of the page is where mature plugins end up, not where you start.

3Write the plugin sentence

Be concrete about who it's for, what it does, and what it replaces. Then fill in this template:

TEMPLATE "This plugin helps [who] [do a task] using [inputs] so they can [outcome] without [today's workaround]."
GOOD "This plugin helps a platform engineer scaffold a new microservice using Beef Co engineering conventions so they can create the standard service files without copying from old repos or asking another team how it's done."

This is 90% of the work. Be specific, and give examples.

4Build the smallest version

Three files is a working plugin. Start with exactly this structure:

scaffold-kit/ .claude-plugin/plugin.json skills/new-service/SKILL.md README.md

The manifest names the plugin. Claude Code discovers the skill from /skills/new-service/SKILL.md:

# .claude-plugin/plugin.json { "name": "scaffold-kit", "description": "Scaffold a new service using Beef Co engineering conventions.", "version": "0.1.0" }

The SKILL.md is the prompt. $ARGUMENTS is what the user types after the command:

# skills/new-service/SKILL.md --- description: Scaffold a microservice to Beef Co spec. --- Create a new service named "$ARGUMENTS". Generate the 5 standard files: ci.yml · Dockerfile · health.ts · README · CODEOWNERS Follow every Beef Co convention. Spell out each assumption before writing files.
Run it: load with claude --plugin-dir ./scaffold-kit, then type /scaffold-kit:new-service payments-api. Claude writes all five files to spec.

5Test it like a stranger

Fresh clone. If it fails, the plugin is not done.
  • claude plugin validate ./scaffold-kit passes clean
  • The skill runs from /scaffold-kit:new-service and produces the files
  • They install from your README alone, with no help from you

6Only then add hooks or MCP

Beginners overbuild. Do not add MCP because it sounds impressive: add it only when pasted-in data is the real bottleneck. Add a hook only when something genuinely must happen automatically. A working one-skill plugin beats an impressive broken one.
