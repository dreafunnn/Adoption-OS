03 How it was built — the interesting decisions 2:00 – 3:30

The plugin has four parts: a hook that logs tool events, a CSV for structured usage data, an agent that harvests patterns from session transcripts, and a skill that formats findings for executives.

The decision I'm most glad I made: I built the hook before I built anything else, and had Claude Code write a test harness that fired fake tool events through it. That's what caught the two bugs that would have silently broken everything in production.

🐛
Bug 1 — Wrong field name. The Claude Code runtime sends tool_name, not tool. Using the wrong name logged every row as unknown. 800 engineers' worth of data that tells you nothing.
🐛
Bug 2 — Four error shapes. Bash failures send isError: true. Edit failures send is_error: true. Different field, different casing. Success detection needed four cases or it marked every infra failure as success.

Where I had to steer Claude Code: The harvester kept ending every analysis with "Would you like me to draft a playbook?" That sounds helpful, but inside a demo it's noise — it breaks the flow and makes it look like the tool can't finish a task on its own.

I had to be explicit: output ends after the anti-patterns section, do not offer next steps, do not ask follow-up questions. That instruction is in the agent's system prompt now. Knowing when "helpfulness" is actually a bug — that's the job.

04 Walking a customer through building their own 3:30 – 4:30

The repo includes a guide called BUILD_YOUR_OWN.md. The pitch: if you can describe a repeated workflow in plain English, you can probably ship a plugin in an afternoon.

1 min
Name the workflow

What's the one thing your best engineers do that your median engineers don't? Write it down. That's your agent prompt.

5 min
Validate before you write logic

Run claude plugin validate . on the stub. Catch structural errors early. We found the agents field pointing to a directory instead of a file before any code existed.

30 min
Write the hook test harness first

Fire fake tool events through logger.sh and confirm your CSV has real data. This is the step most people skip. It's the step that catches the bugs that would otherwise live in production for weeks.

The three things that are actually hard — and where I'd tell any customer to slow down:

  • 1 The hook payload schema is underdocumented. tool_name vs tool will silently break your logging.
  • 2 Success detection has four shapes across tools. Get one wrong and your failure metrics are wrong.
  • 3 CSV formula injection is a real attack surface if engineers are running untrusted code.

All three are handled in this repo. The guide explains why, not just what.
